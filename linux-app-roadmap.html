从 Ubuntu 终端出发，只学 Linux 应用层

嵌入式 Linux 应用开发学习路线

这份路线默认你已经完成 Ubuntu 安装并能打开终端，现在直接从“会用 Linux + 会写 C 程序”开始，最终走到“交叉编译 ARM 程序 + 传到开发板 + 运行控制 LED、按键、串口、网络”。整条路线只关注应用层，不碰内核驱动、不学裸机，强调每天 2 到 3 小时、4 到 6 周、全程实操优先。

Ubuntu 熟练操作 C 与 Makefile 交叉编译 NFS / SCP 传输 设备文件编程 上板运行
起点 你已经打开 Ubuntu，可以直接进入命令行训练和 C 编程
终点 独立完成“写程序 → 交叉编译 → 下载开发板 → 控制硬件”

学习总览

目标不是成为 Linux 理论专家，而是把一条真实的嵌入式应用开发链路完整跑通。你只要把下面每个阶段都敲过、改过、跑过，后面再看资料就会非常快。

总周期 4 到 6 周，每天 2 到 3 小时，周末可用来补实操。
主语言 C 语言为主，Linux 命令行为主，Makefile 必须会写。
开发环境 Ubuntu 20.04 / 22.04，开发板用商家现成 Linux 镜像。
最后成果 至少完成 LED、按键、串口 3 个上板小项目。

六阶段地图

先在 PC 端把 Linux 和 C 写顺，再切换到 ARM 交叉编译，最后把程序传到开发板。每一段都要有产出，不要只看不敲。

阶段 0

环境起步

在 Ubuntu 建好学习目录，装好编译与调试工具，确认系统状态正常。

阶段 1

Linux 基础

熟练命令、目录、权限、网络、Vim、gcc，脱离 Windows 操作习惯。

阶段 2

C 与 Makefile

掌握嵌入式够用的 C 语法、文件 IO、系统调用、多文件工程和 Makefile。

阶段 3

交叉编译

理解 x86 和 ARM 的差别，配置交叉编译器并生成 ARM 可执行文件。

阶段 4

开发板部署

烧录镜像、串口登录、配置网络、打通 NFS 或 SCP 文件传输。

阶段 5-6

应用编程与实战

围绕 /dev/xxx 进行设备编程，并完成 LED、按键、串口三个完整项目。

Linux 终端指令示例大全

这一节可以当作你在 Ubuntu 学习期间的随手查询区。每组命令都配了“文字说明 + 具体使用场景 + 示例写法”，不只是告诉你命令名，而是告诉你什么时候该想到它。

目录与定位

先搞清楚自己在哪里

你每天打开终端后的第一件事，通常不是马上编译，而是确认当前目录、切换到项目目录、看看里面有什么。

pwd

显示当前所在目录的完整路径，避免你在错误位置创建文件或执行命令。

使用场景：你刚打开终端，或者切换了很多次目录，不确定现在到底在哪。

ls / ls -l / ls -a

查看当前目录内容；-l 看详细信息；-a 包含隐藏文件。

使用场景：查看当前工程里有哪些源码、脚本、Makefile，或者确认隐藏配置文件是否存在。

cd

切换目录，配合相对路径和绝对路径使用，是最常用的移动方式。

使用场景：从家目录切到项目目录、从源码目录切到交叉编译目录、从一个阶段切到另一个阶段。

bash
pwd
ls
ls -l
ls -a
cd ~/linux_app_learning
cd 03_cross
cd ..
创建与整理

新建目录、复制文件、重命名文件

做嵌入式开发时，你会频繁创建项目目录、复制模板、重命名源码和备份版本，这一组命令用得非常密集。

mkdir / mkdir -p

创建目录；-p 可以一次创建多层目录，不会因为父目录不存在而失败。

使用场景：建立阶段目录、项目目录、共享目录或者按模块拆分源码目录。

cp / cp -r

复制文件或目录；复制目录时需要加 -r。

使用场景：复制 LED 项目模板，快速得到按键项目模板；备份 Makefile 旧版本。

mv

移动文件，也常用于重命名文件或目录。

使用场景：把 hello.c 改成 led.c，或者把编译结果移动到单独输出目录。

bash
mkdir led_project
mkdir -p 06_projects/{led,key,uart}
cp hello.c led.c
cp -r led key_backup
mv hello.c hello_old.c
mv key_backup 06_projects/
删除与清理

删除文件前先确认目标

Linux 下删除通常没有“回收站”可救，所以这组命令一定要慢一点、确认后再执行。

rm

删除单个文件，适合清理临时输出、旧日志、错误生成物。

使用场景：删掉多余的可执行文件、测试日志或临时文本。

rm -r / rm -rf

递归删除目录；-f 表示强制执行，危险性更高。

使用场景：清理整个测试目录、删除错误创建的工程副本。执行前最好先用 pwd 和 ls 确认位置。

clear

清空当前终端屏幕，不影响文件和程序，只是让输出更清爽。

使用场景：前面输出太多，想把当前编译或运行结果看得更清楚。

bash
rm hello
rm data.txt
rm -r old_demo
rm -rf build
clear
查看文件内容

快速看配置、源码、日志

你不可能每次都用编辑器打开文件。很多时候只是快速看头几行、尾几行、或者整份小文件内容。

cat

直接输出整个文件内容，适合小文件，例如配置片段、简单源码、日志结果。

使用场景：看 Makefile、看一行配置、确认某个文本文件已写入成功。

head / tail

分别查看文件开头和结尾内容，适合大日志或长文件。

使用场景：查看启动日志开头、查看程序报错最后几行、查看内核打印最新内容。

less

分页查看大文件，可上下翻页搜索，比直接 cat 更适合长文本。

使用场景：看长篇日志、厂商脚本文本、很长的配置文件。

bash
cat Makefile
head -n 20 /etc/hosts
tail -n 30 build.log
less /var/log/syslog
查找与过滤

找文件、找关键字、定位报错

嵌入式开发里最怕的不是报错，而是找不到报错出现在哪。学会查找，你定位问题的速度会快很多。

find

按文件名、路径、类型等条件查找文件。

使用场景：找某个源码文件、找交叉编译器位置、找系统里的串口相关设备。

grep / grep -n

在文件中查找关键字；-n 会显示行号，方便回到源码定位。

使用场景：搜索报错信息、搜索函数名、搜索配置项、搜索日志中的关键词。

| 管道

把前一个命令的输出交给后一个命令继续处理，是 Linux 命令组合的核心思想。

使用场景：输出太多时先列出来，再用过滤命令缩小范围。

bash
find ~/linux_app_learning -name "*.c"
find /dev -name "tty*"
grep -n "main" hello.c
grep -n "error" build.log
ls /dev | grep tty
权限管理

程序不能运行时先看权限

很多“程序打不开”“脚本运行不了”“设备文件没法访问”的问题，第一反应就应该检查权限。

chmod

修改文件权限，最常见的是给脚本或程序增加执行权限。

使用场景：交叉编译后的程序传到开发板后，执行前需要 chmod +x。

ls -l

查看文件详细权限、属主、大小和时间信息，是权限排查的起点。

使用场景：确认当前文件是否可执行、当前用户是否具备读写权限。

sudo

以管理员权限运行命令，用于安装软件、修改系统配置、重启服务等操作。

使用场景：安装 NFS、编辑 /etc/exports、挂载系统级目录、查看受限日志。

bash
ls -l led_demo
chmod +x led_demo
chmod 755 run.sh
sudo apt update
sudo vim /etc/exports
压缩与解压

处理工具链、资料包、备份包

厂商资料、交叉编译器、示例工程经常是压缩包。不会解压就没法继续下一步。

tar -xzvf

解压 .tar.gz 压缩包，是 Linux 下最常见的解压方式之一。

使用场景：解压交叉编译器、板卡资料包、示例工程代码包。

tar -czvf

把目录或文件打包压缩，方便备份、分享或上传。

使用场景：把当前工程打包给同学、备份一个稳定可用版本。

unzip

用于解压 .zip 格式文件，一些 Windows 环境导出的资料常会用到。

使用场景：下载到的厂商资料是 zip 包时，先解压再进入目录查看。

bash
tar -xzvf toolchain.tar.gz
tar -xvf rootfs.tar.xz
tar -czvf led_project_backup.tar.gz led_project
unzip vendor_docs.zip
编译与运行

写完代码后最常用的一组命令

从 hello.c 到 LED 项目，核心动作始终是“编译、执行、确认输出”，只是编译器从 gcc 换成了交叉编译器。

gcc

在 PC 上编译 x86 程序，适合前期练习 C 语言和 Linux 文件 IO。

使用场景：先在 Ubuntu 本机验证逻辑是否正确，再切换到 ARM 编译。

make

读取 Makefile 自动完成编译，适合多文件工程和重复构建。

使用场景：项目包含多个源文件时，用一条命令完成所有编译和链接。

file

查看文件类型，能快速确认可执行文件到底是 x86 还是 ARM 架构。

使用场景：程序在开发板上运行失败时，先确认是否编错架构。

bash
gcc hello.c -o hello
./hello
make
make clean
file hello
file led_demo
进程与运行状态

程序卡住、占用高、后台没退出时用它们

你的程序跑起来以后，也要知道它是否真的在运行、是否占 CPU、是否需要手动结束。

ps / ps aux

查看当前系统进程信息，确认程序是否启动成功。

使用场景：串口程序跑在后台后，确认它还在不在；查看 ssh、nfs 服务是否运行。

top

实时查看系统资源占用，包括 CPU、内存和进程变化。

使用场景：程序进入死循环导致 CPU 很高，或者你想观察系统负载。

kill / kill -9

结束指定进程；-9 表示强制终止，通常在程序无响应时使用。

使用场景：测试程序没写退出条件，终端被占住或者后台进程一直不结束。

bash
ps aux | grep led_demo
ps aux | grep ssh
top
kill 12345
kill -9 12345
网络与远程登录

开发板互联必须会的命令

只要进入开发板阶段，这组命令的使用频率会非常高。能否顺利传文件、远程登录，取决于你是否熟练它们。

ip a

查看网卡和 IP 地址，判断 Ubuntu 和开发板是否处在同一网段。

使用场景：配置虚拟机网络后，先确认本机地址；开发板连网后确认拿到的地址。

ping

测试网络是否可达，是最基础的连通性检查手段。

使用场景：Ubuntu 访问开发板前，先 ping 一下；开发板访问 Ubuntu 做 NFS 挂载前也先测试。

ssh

远程登录另一台 Linux 设备，进入 shell 直接执行命令。

使用场景：不用串口也能管理开发板，适合网络已经打通后的日常开发。

scp

通过 SSH 安全复制文件，是最常用的上板传程序方式之一。

使用场景：把交叉编译好的可执行文件传到开发板的 /root 或指定目录。

bash
ip a
ping -c 4 192.168.1.100
ssh root@192.168.1.100
scp led_demo root@192.168.1.100:/root/
scp -r 06_projects root@192.168.1.100:/root/
系统信息与设备

确认架构、内核、设备节点

嵌入式开发经常需要先确认系统到底是什么架构、内核版本是什么、设备文件是否存在，然后才好判断编译和运行问题。

uname -a

查看内核和架构信息，是判断系统身份的最快方法之一。

使用场景：确认当前是 Ubuntu 主机还是开发板，确认板子是不是 ARM 架构。

dmesg

查看内核启动和设备日志，对串口识别、USB 转串口接入、驱动报错很有帮助。

使用场景：插入 USB 转 TTL 后想确认系统分配了哪个 /dev/ttyUSB*。

ls /dev

查看设备节点目录内容，是找 LED、按键、串口设备名的基础动作。

使用场景：代码里要操作 /dev/led0 或 /dev/ttyS1 之前，先确认它真的存在。

bash
uname -a
dmesg | tail
ls /dev
ls /dev/tty*
ls /dev/led*
软件安装与服务

安装工具和重启系统服务

当你需要安装交叉编译辅助工具、网络工具或者 NFS 服务时，这组命令会直接用到。

apt update / apt install

更新软件包索引并安装软件，是 Ubuntu 最基础的软件管理方式。

使用场景：安装 build-essential、vim、nfs-kernel-server、minicom。

systemctl status

查看服务状态，确认服务是否运行正常。

使用场景：NFS 配好后，检查服务有没有成功启动。

systemctl restart

重新启动服务，让配置修改立即生效。

使用场景：修改了 /etc/exports 后，重启 NFS 服务加载新配置。

bash
sudo apt update
sudo apt install -y build-essential vim minicom nfs-kernel-server
sudo systemctl status nfs-kernel-server
sudo systemctl restart nfs-kernel-server
重定向与日志

保存输出、追加日志、临时记录结果

编译输出、程序运行信息、错误日志最好留痕，这样你回头排查问题时会轻松很多。

>

把命令输出写入文件，如果文件已存在会覆盖原内容。

使用场景：保存一次编译输出，或者把目录清单导出成文本。

>>

追加输出到文件末尾，不会覆盖原内容。

使用场景：持续记录实验日志、测试结果、运行过程中的提示信息。

2>

单独重定向错误输出，便于把错误信息保存到专门文件。

使用场景：编译失败时把错误信息单独收集出来，后续分析更方便。

bash
ls -l > file_list.txt
echo "test begin" >> run.log
make > build.log 2> build_error.log
./led_demo 1 >> run.log
帮助与自查

忘了参数时不要硬背，先查

Linux 命令很多，不需要一次性把所有参数都背下来。更重要的是形成“忘了就立刻查”的习惯。

man

查看命令手册，是最标准的帮助入口。

使用场景：不确定 tar、grep、chmod 某个参数是什么意思时。

--help

很多命令支持快速帮助参数，比完整手册更适合临时查基础写法。

使用场景：只想快速看常用参数，不想翻很长的 man 手册。

history

查看你之前输入过的命令，适合回忆某条好用但忘记了的指令。

使用场景：昨天成功挂载过 NFS，今天忘了具体命令，可以先从历史记录里翻。

bash
man grep
tar --help
chmod --help
history
history | grep scp
阶段 0 · 0.5 天

从“已打开 Ubuntu”正式起步

你已经进入 Ubuntu，所以这里不再讲安装系统，而是先把学习目录和基础工具准备好。这个阶段完成后，后面所有代码、命令、Makefile 都会有统一位置，避免学习过程越来越乱。

建立统一学习目录

建议产出：得到一个固定的学习工作区。

建议产出：Ubuntu 具备编译和基础调试能力。

bash

阶段 1 · 第 1~2 周

PC 端 Linux 基础必须练熟

这一阶段最重要的不是“知道命令是什么”，而是让终端操作变成下意识动作。只要你能顺手建目录、找文件、改权限、写 hello.c、编译运行，后面的节奏就会非常顺。

阶段目标：脱离 Windows 习惯，能在 Ubuntu 终端里高频完成查找、编辑、编译、运行。
Step 1.1

练熟目录与文件命令

这一组命令是所有后续工作的基本动作。建议自己新建一个练习目录，反复敲 3 遍以上。

重点命令：ls、cd、pwd、mkdir、cp、mv、rm、find、grep。

bash
cd ~/linux_app_learning/01_linux
mkdir demo_dir
cd demo_dir
touch a.txt b.txt
ls -l
cp a.txt a_copy.txt
mv b.txt notes.txt
find . -name "*.txt"
grep -n "hello" *.txt
cd ..
rm -rf demo_dir
Step 1.2

理解 Linux 目录结构

你不用把每个目录都背成百科全书，但一定要知道程序、配置、设备文件分别放哪里。后续找串口、找设备节点、找配置文件都会用到。

重点理解：/dev、/etc、/home、/usr、/bin。

bash
ls /
ls /dev | head

Step 2.2

掌握标准 IO 和 Linux 文件 IO

后面控制设备的核心思想就是“设备也是文件”。所以你一定要会 open、read、write、close。

先在 PC 上操作普通文本文件，理解调用方式。

c
#include <fcntl.h>
#include <stdio.h>
#include <string.h>
#include <unistd.h>

int main(void)
{
    int fd;
    char buf[64] = {0};
    const char *msg = "hello file io\\n";

    fd = open("data.txt", O_CREAT | O_RDWR | O_TRUNC, 0644);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    write(fd, msg, strlen(msg));
    lseek(fd, 0, SEEK_SET);
    read(fd, buf, sizeof(buf) - 1);
    printf("read from file: %s", buf);
    close(fd);
    return 0;
}
Step 2.3

对比 stdio 与系统调用 IO

应用开发里两种方式都会遇到。普通文本和日志常用 fopen，设备节点和串口通常更接近系统调用风格。

形成直觉：设备文件优先想 open/read/write。

c
#include <stdio.h>

int main(void)
{
    FILE *fp = fopen("note.txt", "w+");
    char buf[64] = {0};

    if (fp == NULL) {
        perror("fopen");
        return 1;
    }

    fprintf(fp, "stdio demo\\n");
    rewind(fp);
    fgets(buf, sizeof(buf), fp);
    printf("buf = %s", buf);
    fclose(fp);
    return 0;
}

把程序拆成多文件

建议结构：main.c、file_ops.c、file_ops.h。

c
/* file_ops.h */

#define FILE_OPS_H
int write_text(const char *filename, const char *text);
#endif

/* file_ops.c */
#include "file_ops.h"
#include <fcntl.h>
#include <string.h>
#include <unistd.h>

int write_text(const char *filename, const char *text)
{
    int fd = open(filename, O_CREAT | O_WRONLY | O_TRUNC, 0644);
    if (fd < 0) return -1;
    write(fd, text, strlen(text));
    close(fd);
    return 0;
}
Step 2.5

手写第一个 Makefile

Makefile 是嵌入式开发的重中之重。你以后切换交叉编译器、加源文件、改头文件，都会靠它统一管理。

做到一条 make 命令编译成功，一条 make clean 清理干净。

makefile
CC = gcc
CFLAGS = -Wall -g
TARGET = file_demo
OBJS = main.o file_ops.o

$(TARGET): $(OBJS)
	$(CC) $(CFLAGS) -o $(TARGET) $(OBJS)

main.o: main.c file_ops.h
	$(CC) $(CFLAGS) -c main.c

	$(CC) $(CFLAGS) -c file_ops.c

clean:
	rm -f $(TARGET) $(OBJS)

Step 5.3

理解 open / read / write / ioctl 组合

简单设备常常直接 read / write 就够了；复杂设备可能需要 ioctl 做额外控制。你先知道这个层次关系即可。

形成思路：先 open，再根据驱动接口决定 read/write/ioctl。

c
#include <fcntl.h>
#include <stdio.h>
#include <sys/ioctl.h>
#include <unistd.h>

int main(void)
{
    int fd = open("/dev/led0", O_RDWR);
    int cmd = 1;

    if (fd < 0) {
        perror("open");
        return 1;
    }

    write(fd, "1", 1);
    ioctl(fd, cmd, 0);
    close(fd);
    return 0;
}
Step 5.4

学会基本串口编程

串口程序的核心不是“会收发字符”这么简单，而是你要会用 termios 配置波特率、数据位、停止位和校验位。

至少掌握 115200、8N1 这种最常见配置。

c
#include <fcntl.h>
#include <stdio.h>
#include <string.h>
#include <termios.h>
#include <unistd.h>

int main(void)
{
    int fd = open("/dev/ttyS1", O_RDWR | O_NOCTTY);
    struct termios tty;

    if (fd < 0) {
        perror("open");
        return 1;
    }

    tcgetattr(fd, &tty);
    cfsetispeed(&tty, B115200);
    cfsetospeed(&tty, B115200);
    tty.c_cflag |= (CLOCAL | CREAD);
    tty.c_cflag &= ~CSIZE;
    tty.c_cflag |= CS8;
    tty.c_cflag &= ~PARENB;
    tty.c_cflag &= ~CSTOPB;
    tcsetattr(fd, TCSANOW, &tty);

    write(fd, "hello uart\\n", 11);
    close(fd);
    return 0;
}
Step 5.5

了解简单线程、定时和网络

这些内容不是现阶段主线，但你至少要知道基础写法，后面做轮询按键、周期采样、TCP 通信时会用到。

会看懂线程创建和 socket 初始化的大体结构。

c
#include <arpa/inet.h>
#include <pthread.h>
#include <stdio.h>
#include <sys/socket.h>
#include <unistd.h>

void *worker(void *arg)
{
    while (1) {
        printf("thread running\\n");
        sleep(1);
    }
    return NULL;
}

int main(void)
{
    pthread_t tid;
    int sockfd = socket(AF_INET, SOCK_STREAM, 0);

    pthread_create(&tid, NULL, worker, NULL);
    printf("socket fd = %d\\n", sockfd);
    close(sockfd);
    pthread_cancel(tid);
    pthread_join(tid, NULL);
    return 0;
}

项目 2：读取板载按键

按键项目的关键是从 /dev/key0 读取状态，并根据值打印提示。这样你会更直观地理解“读设备文件”的感觉。

结果应为：按下按键时终端出现提示信息。

c
#include <fcntl.h>
#include <stdio.h>
#include <unistd.h>

int main(void)
{
    int fd;
    char key_value;

    fd = open("/dev/key0", O_RDONLY);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    while (1) {
        if (read(fd, &key_value, 1) > 0) {
            if (key_value == '1') {
                printf("key pressed\\n");
            } else {
                printf("key released\\n");
            }
        }
        usleep(100000);
    }

    close(fd);
    return 0;
}
Step 6.5

项目 3：串口收发并回显

第三个项目做完，你就已经不只是“会写个小程序”，而是真正开始具备和外设交互的应用开发能力了。

结果应为：PC 发数据给开发板，开发板接收后再回显回来。

c
#include <fcntl.h>
#include <stdio.h>
#include <termios.h>
#include <unistd.h>

int main(void)
{
    int fd;
    char buf[128];
    int n;
    struct termios tty;

    fd = open("/dev/ttyS1", O_RDWR | O_NOCTTY);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    tcgetattr(fd, &tty);
    cfsetispeed(&tty, B115200);
    cfsetospeed(&tty, B115200);
    tty.c_cflag |= (CLOCAL | CREAD);
    tty.c_cflag &= ~CSIZE;
    tty.c_cflag |= CS8;
    tty.c_cflag &= ~PARENB;
    tty.c_cflag &= ~CSTOPB;
    tcsetattr(fd, TCSANOW, &tty);

    while (1) {
        n = read(fd, buf, sizeof(buf));
        if (n > 0) {
            write(fd, buf, n);
            write(STDOUT_FILENO, buf, n);
        }
    }

    close(fd);
    return 0;
}

可选进阶方向

等 3 个核心项目跑通之后，再往下扩展就会顺很多。此时再学简单 TCP 服务端、蜂鸣器、传感器读取，学习体验会明显更好。

进阶 1

简单 TCP 服务器

开发板监听端口，PC 通过网络连接，发送指令控制 LED 或查询状态。

进阶 2

蜂鸣器控制

如果板卡厂商已经提供设备节点，可以沿用 LED 的写法，快速完成蜂鸣器启停。

进阶 3

环境传感器读取

通过串口、I2C 工具或已有驱动设备节点读取温湿度等数据，再把结果打印或上传。

避坑重点

下面这些坑基本是新手最容易在前 2 周和上板初期踩到的。提前记住，能省很多时间。

重点 1

不要一开始学内核驱动

你当前目标是做应用层闭环。先把设备节点操作、交叉编译、上板流程做顺，收益远高于一开始啃驱动。

重点 2

编译器架构必须匹配

32 位 ARM 和 64 位 ARM 工具链不同。交叉编译器不匹配时，程序即使编译成功，也可能在开发板上无法运行。

重点 3

优先用商家现成镜像

现阶段没必要自己编内核、做 rootfs。先把 Linux 跑起来、能进 shell、能传程序，这样学习效率最高。

重点 4

尽量全程在 Ubuntu 完成

不要总退回 Windows 工具思维。Linux 命令行、ssh、scp、minicom、nfs 这些能力本身就是你要学的内容。

重点 5

设备节点名不要硬背

不同开发板的 LED、按键、串口节点名可能不一样。先 ls /dev 观察，再改代码中的路径。

重点 6

每个阶段都要留下产物

至少保留源码、Makefile、编译命令和运行截图。这样你后面回头复习时会非常轻松。

极简链路

把这条链路刻进脑子里，后面做任何嵌入式 Linux 应用项目，本质上都在重复它。

Ubuntu 写 C 程序 写 Makefile 交叉编译 ARM 程序 NFS / SCP 传开发板 chmod +x 授权 运行程序 控制硬件

如果你能不看资料，独立把这条链路跑通 3 次以上，就说明你已经真正入门嵌入式 Linux 应用开发了。
